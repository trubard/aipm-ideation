# AIPM — 아이디어 검증 워크스페이스 (개인용)

> v2.4 · 2026-06-17 · 이 저장소는 **각자 받아서 자기 아이디어를 검증·구체화하는** 개인 작업 공간이다.
> 아이디어를 **(문제 발굴) → 풀 → 탐색 → 검증 → 정의**의 깔때기로 흘려보내고, 막히면 `my-idea/99_아카이브`로 보낸다.
> Claude Code의 슬래시 명령어 7개(`/초기화`·`/문제발굴`·`/아이디어검증`·`/고객인터뷰`·`/설문조사`·`/아이디어진행`·`/아이디어폐기`)로 단계를 돌린다.

전부 마크다운(`.md`)이라 GitHub 웹에서 바로 읽히고, 원하면 이 폴더를 Obsidian으로 열어도 된다.

## 구조: `structure/`(시스템) vs `my-idea/`(내 작업)

저장소는 두 층으로 나뉜다. **시스템과 내 데이터를 섞지 않기 위해서다.**

- **`structure/`** — 공유·버전관리되는 **빈 골격(시스템)**. 단계 폴더의 골격 + `90_템플릿/`(작성 가이드·빈 양식)을 담는다. **읽기 전용으로 취급**(양식은 여기서 읽기만). `git pull`로 업데이트되는 게 이 폴더다.
- **`my-idea/`** — 내 **실제 아이디어 작업 공간**. `structure/`의 단계 폴더를 복제한 것으로, 모든 산출물이 여기 쌓인다. **`.gitignore`로 공유에서 제외**(개인 데이터는 로컬에만).

가장 쉬운 방법은 **`/초기화`** 명령어다(생성·리셋을 알아서 처리, 리셋은 확인을 받는다). 직접 셸로 해도 된다:

```bash
# 최초 1회 — structure에서 내 작업 공간 만들기
mkdir -p my-idea && cp -R structure/10_아이디어풀 structure/20_탐색중 structure/30_검증정의 structure/99_아카이브 my-idea/

# 전체 초기화 — 내 산출물 싹 지우고 빈 골격으로 리셋 (되돌릴 수 없음)
rm -rf my-idea && mkdir -p my-idea && cp -R structure/10_아이디어풀 structure/20_탐색중 structure/30_검증정의 structure/99_아카이브 my-idea/
```

> 6개 슬래시 명령어는 모두 **산출물을 `my-idea/`에 쓰고, 양식은 `structure/90_템플릿/`에서 읽는다.**

## 빠른 시작

0. (처음이면) **`/초기화`** 로 `my-idea/` 작업 공간을 만든다.
1. 떠오른 아이디어를 [my-idea/10_아이디어풀/아이디어_백로그.md](my-idea/10_아이디어풀/아이디어_백로그.md)에 **한 줄**로 적는다.
   - (아이디어 자체가 막연하면) **`/문제발굴 "<영역>"`** → 외주·리뷰·커뮤니티·검색을 훑어 *미해결 문제*를 6축으로 태깅해 [문제 인박스](my-idea/10_아이디어풀/문제_인박스.md)에 쌓고, 점수 높은 건 백로그 한 줄로 승급한다.
2. **`/아이디어검증 "<한 줄 아이디어>"`** → 문제정의·경쟁탐색·비즈니스 검증을 거쳐 Go/Pivot/Drop 판정이 나온다.
   - (선택) 가장 위험한 가정을 고객에게 직접 물으려면 **`/고객인터뷰`**(왜=정성) / **`/설문조사`**(몇%=정량) → 둘 다 The Mom Test 기반.
3. Go/Pivot이면 **`/아이디어진행 "<아이디어>" "HMW까지"`** → 페르소나·JTBD·Journey·HMW를 이어서 채운다.
4. 게이트를 통과하면 같은 명령으로 **30단계(NorthStar~PRD~피치덱)**까지. 접을 거면 **`/아이디어폐기`**.

## 단계별 깔때기

단계 폴더는 모두 `my-idea/` 아래에 있다(`90_템플릿`만 `structure/`).

```
my-idea/
 10_아이디어풀 ──────────► 20_탐색중 ──► 30_검증정의 ──► (출시/이관)
  문제 인박스 ─► 한 줄        문제정의·페르소나   PRD·MVP·피치덱
  (/문제발굴)    백로그        JTBD·Journey·HMW    NorthStar·Aha·AARRR
                                 │
                                 └──► 99_아카이브 (드롭/보류)
```

| 폴더 | 무엇이 들어가나 | 다음 단계로 가는 게이트 |
|---|---|---|
| **my-idea/10_아이디어풀** | `문제_인박스.md`(발굴한 미해결 문제 풀, `/문제발굴`이 채움) + `아이디어_백로그.md`(**한 줄 아이디어**만, 가볍게 많이) | 문제: 6축 점수 높고 게이트 통과하면 → 백로그 한 줄로 / 아이디어: 사용자·대안·핵심 또렷하면 → 검증 |
| **my-idea/20_탐색중** | 아이디어당 폴더 1개. 문제정의·페르소나·JTBD·Journey·HMW | 문제가 진짜고 크고, HMW에서 쓸 만한 솔루션이 나오면 → 검증·정의 |
| **my-idea/30_검증정의** | 아이디어당 폴더 1개. NorthStar·Aha·AARRR·MVP 스펙·PRD·피치덱 | MVP 가설 검증되면 → 본 개발/스핀아웃 이관 |
| **my-idea/99_아카이브** | 폐기·보류 아이디어 + 死因·교훈·부활 트리거([대장](my-idea/99_아카이브/README.md)). `/아이디어폐기`로 이관 | 부활 트리거 충족 시 10/20으로 |
| **structure/90_템플릿** | `기획_산출물_템플릿.md`(작성 가이드) + `양식/`(빈 양식: **문제발굴(AI BM 6축)**·5Whys·페르소나·JTBD·HMW·NorthStar·Aha·AARRR·PRD·**아이디어검증**·**고객인터뷰/고객설문(Mom Test)**·**폐기기록** 등). 읽기 전용 | — |

## 명령어로 할 수 있는 것

Claude Code 채팅창에 아래를 입력하면 된다. 깔때기 단계마다 쓰는 명령어가 정해져 있다.

| 단계(목적) | 명령어 | 하는 일 | 결과물 |
|---|---|---|---|
| **준비** (작업공간) | `/초기화 [생성\|리셋]` | `structure/` 골격에서 `my-idea/`를 **생성**하거나, 내 산출물을 싹 비우고 **리셋**한다(리셋은 확인 필수, `structure/`는 안 건드림) | `my-idea/` 생성 또는 빈 골격으로 초기화 |
| **문제 발굴** (→10) | `/문제발굴 <영역>` | 한 영역에서 **이미 돈·시간 쓰는 신호**(외주·유료앱 리뷰·커뮤니티·검색)를 WebSearch로 훑어 *미해결 문제*를 줍고, **AI BM 6축**(반복성·데이터비대칭·비용장벽·Why now·방어가능성 + 피처리스크·WTP 게이트)으로 태깅 | `my-idea/10_아이디어풀/문제_인박스.md`에 적재, 상위 후보는 백로그 원석으로 승급 |
| **검증** (10→20) | `/아이디어검증 <한 줄 아이디어>` | 문제정의(페르소나·JTBD·빈도·강도) → **유사 해결방식 탐색**(WebSearch로 경쟁제품 + 유튜브 실검색) → 비즈니스 검증 → **현실 점검 체크리스트(7문항)** → **Go/Pivot/Drop 판정** | `my-idea/20_탐색중/<슬러그>/00_아이디어검증.md` 생성, 백로그 반영. Drop이면 `my-idea/99_아카이브`로 이동 |
| **고객 인터뷰** (검증 보강·정성) | `/고객인터뷰 "<아이디어>"` | 검증의 **"가장 위험한 가정"을 정조준**해 **The Mom Test 기반** 인터뷰 가이드 생성(섭외·과거사실 위주 대본·금지질문·기록·종합). 칭찬이 아니라 *행동·커밋먼트*로 검증. "왜"를 캔다 | 아이디어 폴더에 `인터뷰_가이드.md` 생성, 백로그 반영 |
| **고객 설문** (검증 보강·정량) | `/설문조사 "<아이디어>"` | 같은 가정을 **다수 응답으로 양적 검증**. 과거행동·실제지출·행동신호 문항 + **합격선 사전 설정** + 금지문항 가드레일. 인터뷰 패턴이 "몇 %"인지 잰다 | 아이디어 폴더에 `설문_문항.md` 생성, 백로그 반영 |
| **구체화·확정·PRD** (20→30) | `/아이디어진행 "<아이디어>" [목표]` | **현재 단계를 감지**해 빠진 다음 산출물을 이어서 생성. 20단계는 5Whys→페르소나→JTBD→Journey→HMW, 30단계는 NorthStar→Aha→AARRR→MVP→**1페이저 PRD**→피치덱 | 단계 폴더에 번호 접두사로 저장(예: `02_페르소나.md`, `05_HMW.md`, `10_PRD_1페이저.md`), 백로그 단계 갱신 |
| **폐기** (→아카이브) | `/아이디어폐기 "<아이디어명>" "<폐기 사유>"` | 접은 아이디어를 **사유·교훈·부활 트리거**와 함께 정리. 산출물은 지우지 않고 **이동·보존** | 폴더째 `my-idea/99_아카이브/<이름>/`로 이동 + `00_폐기기록.md`, [아카이브 대장](my-idea/99_아카이브/README.md)·백로그 갱신 |

**`/아이디어진행`의 목표(2번째 인자)** — 없으면 **다음 산출물 1개**만 만들고 멈춘다(검수 후 다시 호출). 어디까지 한 번에 갈지 지정 가능:

```
/아이디어진행 "레시피봇"              # 다음 빠진 산출물 1개만
/아이디어진행 "레시피봇" "HMW까지"     # 20단계 산출물을 HMW까지 연속
/아이디어진행 "레시피봇" "PRD까지"     # 30단계로 승급해 1페이저 PRD까지
/아이디어진행 "레시피봇" "끝까지"      # 피치덱까지 (단, 20→30 승급은 확인을 받는다)
```

> - 세 명령어 모두 인자가 비면 먼저 되묻는다. `/아이디어폐기`는 폴더·산출물을 **삭제하지 않는다**(이동·보존).
> - `/아이디어진행`은 **검증 없이 구체화 금지** — `00_아이디어검증.md`가 없거나 판정이 Drop이면 진행을 멈추고 안내한다. **20→30 직행도 금지** — HMW까지 끝나면 게이트 통과 여부를 보고하고 확인을 받아 승급한다.

**대표 흐름 (검증 → 구체화 → 확정 → 출시/폐기)**

```
/아이디어검증 노인 대상 복약 알림 앱        # my-idea/20_탐색중에 검증 산출물 + 판정
   ├─ Go/Pivot → /아이디어진행 "복약알림" "HMW까지"   # 20단계 구체화
   │              └─ 게이트 통과 → /아이디어진행 "복약알림" "PRD까지"  # 30단계 확정·PRD
   │                                                 └─ MVP 가설 검증 → 본개발/스핀아웃 이관
   └─ Drop     → /아이디어폐기 "복약알림" "수요 신호 약함, 경쟁 과밀"
```

## 작성 규칙

- **한 아이디어 = 한 폴더.** `my-idea/20_탐색중/`·`my-idea/30_검증정의/` 아래에 한글 아이디어명 폴더(공백 대신 붙여쓰기, 예: `장보기도우미`)를 만든다. 진행 상황은 [my-idea/10_아이디어풀/아이디어_백로그.md](my-idea/10_아이디어풀/아이디어_백로그.md) 표에서 한눈에 본다.
- **한 줄 아이디어 양식**: `[누가—구체조건]`가 `[지금 하는 대안/습관]` 대신 `[핵심 한 가지]`로 `[결과—가능하면 숫자]`. 원석은 양식 안 맞아도 일단 적는다.
- **산출물 형식·순서·체크리스트**는 [structure/90_템플릿/기획_산출물_템플릿.md](structure/90_템플릿/기획_산출물_템플릿.md)를 따른다. 빈 양식은 [structure/90_템플릿/양식/](structure/90_템플릿/양식/). 명령어를 안 쓰고 수동으로 채워도 된다.
- **일관성**: 같은 아이디어 안에서는 사용자 이름·핵심 숫자를 **고정**한다(예: 사용자=김OO 34세). 문서 사이에서 이름·숫자가 흔들리지 않게.
- **파일 규칙**: 파일명은 양식 번호 접두사(`02_페르소나.md`) 또는 `산출물종류_아이디어명.md`. 문서 상단에 `> vX.Y · 날짜` 메타 한 줄을 유지한다.
- **숫자로 끝낸다**: "편하게/많이/좋게"는 산출물이 아니다. 조건·기간·수치를 붙인다.

## 업데이트 & 마이그레이션

이 저장소는 **각자 받아 쓰는 개인용 워크스페이스**다. "배포"는 `git push`, "업데이트"는 `git pull`. 변경 이력은 [CHANGELOG.md](CHANGELOG.md) 참고.

도구(명령어·양식) 변경은 대부분 **추가 위주(additive)**라 pull하면 그냥 새 기능이 생긴다. **내 작업물(`my-idea/`)은 `.gitignore`라 pull 충돌이 없다.** 다만 두 가지만 주의:

| 상황 | 어떻게 |
|---|---|
| **새 기능 받기** | `git pull` — `.claude/commands/`·`structure/90_템플릿/`이 갱신된다. `my-idea/`의 내 산출물은 건드리지 않으며 그대로 동작(명령어가 섹션 번호가 아니라 *내용*으로 찾으므로 옛 파일도 호환). |
| **structure/my-idea 분리(v2.3~)** | 단계 폴더가 `structure/`(빈 골격) + `my-idea/`(내 작업, gitignore)로 나뉘었다. clone/pull 후 `my-idea/`가 없으면 위 "최초 1회" 명령으로 만든다. **이전 버전에서 루트에 만든 아이디어 폴더가 있다면** 해당 폴더를 `my-idea/<단계>/` 아래로 옮긴다. |
| **옛 산출물에 새 항목이 없음** | 양식이 바뀌어도 *이미 만든* `00_아이디어검증.md`엔 새 항목(예: §4 현실 점검 체크리스트)이 자동 추가되지 않는다. 필요하면 그 아이디어에 `/아이디어검증`을 재실행하거나, 양식의 해당 섹션만 수동으로 덧붙인다(강제 아님). |

> 전체를 빈 상태로 되돌리려면 위 "전체 초기화" 명령으로 `my-idea/`만 리셋하면 된다(`structure/` 시스템은 유지된다).
